# 代码复用思考指南

在这个 SDK 里，复用的重点不是“抽一个大而全框架”，而是避免把同一份协议知识散落到多个层。

## 开始前先问

- 这个字段映射、错误判断或 URL 归一化，是否已经在别的模块做过一次？
- 新逻辑是供应商特有，还是所有 Provider 都会共享？
- 我是在复用公开模型，还是又偷偷造了一份平行结构体？

## 当前仓库最值得优先复用的点

- `include/core/` 里的消息、请求、响应模型，而不是在 Provider 里自建 JSON 中间结构。
- `AIClient` 的 Provider 选择入口，而不是让示例或调用方自己拼不同 Provider 实例。
- `DeepSeekProvider` 内部的请求头、URL 和请求体辅助函数，而不是多处重复拼接。
- `SSEParser` 的事件解释能力，而不是在上层重新手写 `data:` 解析。
- `tests/core/`、`tests/http/` 现有断言风格，而不是每新增一个行为都只写冒烟测试。

## 看到这些信号时，优先停下来复用

- 两处以上在处理同一个 JSON 字段名，例如 `model`、`messages`、`tool_calls`。
- 两处以上在判断同一个错误条件，例如 API Key 缺失、Provider 名非法。
- 示例和测试里出现两套不同的推荐调用方式。
- 新逻辑需要重复做 URL 归一化、环境变量装载或 SSE 事件切分。

## 当前仓库里不该强行抽象的地方

- 只有一个实现的特有行为，不要先抽成庞大的“通用 Provider 工厂”。
- 只出现一处的简单数据搬运，不要为了“看起来统一”引入过度模板或宏。
- 预留目录里的未来能力，例如 `agent/`、`trace/`，没有真实重复前不要先做框架。

## 落地检查

- 搜索现有实现时，优先看 `include/core/`、`src/provider/`、`src/http/` 与对应测试。
- 如果决定不复用，原因必须是职责不同或语义不同，而不是“没看到马上可复制的函数”。
