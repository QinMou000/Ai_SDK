# 日志规范

## 当前现状

仓库已经在 `CMakeLists.txt` 中链接 `spdlog::spdlog`，配置模型 `Config` 也有 `enable_trace` 字段，但当前生产代码里还没有形成统一日志落点：

- `src/` 目录没有稳定的 `spdlog` 调用模式。
- 示例和测试依赖返回值、异常和事件断言，而不是解析日志输出。

因此，当前最重要的规则是“不要引入失控的临时日志”，而不是发明一套仓库里还不存在的日志框架。

## 当前规则

### 1. 不要随手加入 `std::cout` / `printf` 调试输出

- SDK 是库代码，不是命令行程序。
- 随意向标准输出写内容会污染调用方进程，尤其是流式输出场景。

### 2. 如果必须补日志，先走统一开关

- 调试链路优先使用 `Config::enable_trace` 控制的结构化 Trace，不新增散落的布尔开关。
- 日志行为必须可关闭，默认不能影响正常输出。

### 3. 结构化 Trace 不是日志

- Trace 数据通过 `TraceSession::snapshot()` 或 `toJson()` 由调用方主动读取，SDK 不自动打印。
- 日志不得成为 Trace 的数据源，测试也不得通过解析日志字符串验证步骤。
- Trace 的白名单、脱敏和异常隔离以 [Trace 契约](./trace-contracts.md) 为唯一来源。

### 4. 协议原文与敏感头信息不能直接打印

- `DeepSeekProvider` 请求头里包含 `Authorization: Bearer ...`。
- 即使开启诊断，也不能把完整 API Key、原始鉴权头或用户敏感消息明文输出到日志。

### 5. 日志只记录边界事实，不承担业务分支

- 可以记录请求开始、响应状态码、SSE 事件类型数量、重试决策等。
- 不要通过“看日志字符串”驱动程序分支，也不要让测试依赖日志文本。

## 适合记录的内容

- 选择了哪个 Provider、哪个模型。
- HTTP 状态码、超时、是否进入流式模式。
- SSE 是否收到 `Done`、是否出现错误事件。
- 本地配置装载路径与是否命中 `.env`。

## 不适合记录的内容

- 完整 API Key 或完整 `Authorization` 头。
- 整段原始用户消息与工具参数，除非调用方明确提供脱敏策略。
- 高频每 token 一条的无节制日志。

## 未来扩展建议

如果后续要正式启用 `spdlog`：

- 先在一个集中位置定义 logger 生命周期。
- 为 `trace`、`info`、`warn`、`error` 明确用途。
- 同步更新本规格与测试策略，确保日志不会变成隐式接口。
