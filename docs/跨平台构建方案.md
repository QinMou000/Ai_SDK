# AI SDK 依赖库与跨平台构建方案

## 1. 依赖选择原则

本项目目标是纯 C++ SDK，并且要同时支持 Windows 和 Linux。因此依赖选择遵循四个原则：

1. **跨平台优先**：依赖库必须能在 Windows / Linux 上稳定构建。
2. **CMake 友好**：所有依赖都能通过 `find_package()` 或 CMake target 链接。
3. **vcpkg 管理**：统一用 `vcpkg.json` 声明依赖，避免手动配置 include/lib 路径。vcpkg 官方推荐大多数项目使用 manifest mode，即通过项目里的 `vcpkg.json` 声明依赖，并为每个项目单独安装依赖，避免不同项目依赖冲突。
4. **第一版少依赖**：能自己写的简单模块先自己写，比如 SSE Parser、ToolCall Parser、Skill 目录扫描。

## 2. 第一版推荐依赖

### 2.1 必选依赖

| 能力      | 推荐库           | 用途                                        |
| ------- | ------------- | ----------------------------------------- |
| 构建系统    | CMake         | 跨平台构建                                     |
| 包管理     | vcpkg         | 管理第三方 C++ 依赖                              |
| HTTP 请求 | cpr           | 调用 DeepSeek / MiniMax API                 |
| JSON 解析 | nlohmann/json | 请求体、响应体、ToolCall 参数解析                     |
| 日志      | spdlog        | SDK 内部日志                                  |
| 单元测试    | GoogleTest    | 测试 Message、Provider、ToolCall、ToolRegistry |

### 2.2 为什么这样选

#### CMake

CMake 是这个项目的构建核心。建议项目根目录放 `CMakeLists.txt` 和 `CMakePresets.json`。

`CMakePresets.json` 可以把常用构建配置写进项目里，方便 Windows、Linux、CI 使用同一套命令。CMake 官方文档说明，`CMakePresets.json` 用于共享项目级构建配置，`CMakeUserPresets.json` 用于开发者本地私有配置，前者应该提交到版本控制，后者不应该提交。

#### vcpkg

vcpkg 用于解决跨平台依赖安装问题。

项目里放一个 `vcpkg.json`，Windows 和 Linux 都根据这个文件安装依赖。这样别人 clone 项目后，不需要手动找库、配路径、改 CMake。

#### cpr

cpr 是一个基于 libcurl 的 C++ HTTP 请求库，语法比直接用 libcurl 简单。cpr 官方 README 说明它是 “C++ Requests”，是一个围绕 libcurl 的简单封装，并且当前主线版本使用 C++17。

本项目第一版用 cpr 足够：

* POST JSON 请求
* 设置 Header
* 设置 Bearer Token
* 设置 Timeout
* 读取响应状态码
* 处理流式响应回调

底层 HTTPS、OpenSSL、curl 的平台差异交给 vcpkg 处理。

#### nlohmann/json

nlohmann/json 用于所有 JSON 解析和构造。

它的 CMake 集成很友好，官方文档明确提供 `nlohmann_json::nlohmann_json` 这个 CMake target，可以直接 `target_link_libraries()` 链接。

本项目里它负责：

* 构造模型请求 JSON。
* 解析模型响应 JSON。
* 解析 ToolCall arguments。
* 读取 `config.json`。
* 读取 `tools.json`。
* 后续读取 MCP JSON-RPC 消息。

#### spdlog

spdlog 用来做日志。它支持 header-only 或编译模式，也支持控制台、文件、异步日志、Windows debugger、Windows event log 等输出目标。

本项目第一版用它做：

* Provider 请求日志。
* HTTP 错误日志。
* JSON 解析错误日志。
* ToolCall 解析日志。
* Tool 执行日志。

#### GoogleTest

GoogleTest 用于单元测试。官方 quickstart 明确支持 CMake，并要求兼容 C++17 的编译器。

第一版至少要测：

* `Message` 序列化。
* `ChatRequest` 转 JSON。
* `ToolSchema` 转 JSON Schema。
* `ToolCallParser` 解析 DeepSeek / MiniMax 响应。
* `ToolRegistry` 注册和执行工具。
* `SSEParser` 解析流式片段。

## 3. 第一版不建议引入的依赖

| 依赖           | 暂不引入原因                      |
| ------------ | --------------------------- |
| Boost        | 太重，第一版没必要                   |
| Qt           | 会让项目像 Qt 应用，不像通用 SDK        |
| SQLite       | 第一版 SDK 不急着做持久化             |
| yaml-cpp     | 配置先用 JSON，减少依赖              |
| protobuf     | MCP / ToolCall 都是 JSON，不需要  |
| OpenSSL 直接调用 | 由 cpr/libcurl 间接处理即可        |
| stduuid      | 第一版可以自己生成简单 trace_id        |
| fmt          | spdlog 已经会处理格式化，第一版不单独用 fmt |

## 6. 顶层 CMakeLists.txt

```cmake
cmake_minimum_required(VERSION 3.23)

project(ai_sdk
    VERSION 0.1.0
    LANGUAGES CXX
)

option(AISDK_BUILD_EXAMPLES "Build examples" ON)
option(AISDK_BUILD_TESTS "Build tests" ON)

set(CMAKE_CXX_STANDARD 17)
set(CMAKE_CXX_STANDARD_REQUIRED ON)
set(CMAKE_CXX_EXTENSIONS OFF)

find_package(cpr CONFIG REQUIRED)
find_package(nlohmann_json CONFIG REQUIRED)
find_package(spdlog CONFIG REQUIRED)

add_library(ai_sdk
    src/core/Config.cpp
    src/core/Message.cpp
    src/provider/DeepSeekProvider.cpp
    src/provider/MiniMaxProvider.cpp
    src/http/HttpClient.cpp
    src/http/SSEParser.cpp
    src/tool/ToolRegistry.cpp
    src/tool/ToolExecutor.cpp
    src/tool/ToolCallParser.cpp
    src/trace/TraceRecorder.cpp
    src/AIClient.cpp
)

target_include_directories(ai_sdk
    PUBLIC
        ${CMAKE_CURRENT_SOURCE_DIR}/include
)

target_link_libraries(ai_sdk
    PUBLIC
        cpr::cpr
        nlohmann_json::nlohmann_json
        spdlog::spdlog
)

if (MSVC)
    target_compile_options(ai_sdk PRIVATE /W4 /utf-8)
else()
    target_compile_options(ai_sdk PRIVATE -Wall -Wextra -Wpedantic)
endif()

if (AISDK_BUILD_EXAMPLES)
    add_subdirectory(examples)
endif()

if (AISDK_BUILD_TESTS)
    enable_testing()
    add_subdirectory(tests)
endif()
```

## 8. Windows 构建方式

### 8.1 安装前置环境

Windows 上建议使用：

* Visual Studio 2022 / Build Tools for Visual Studio 2022
* CMake
* Ninja
* Git
* vcpkg

### 8.2 安装 vcpkg

```powershell
git clone https://github.com/microsoft/vcpkg.git C:\dev\vcpkg
cd C:\dev\vcpkg
.\bootstrap-vcpkg.bat
```

设置环境变量：

```powershell
setx VCPKG_ROOT C:\dev\vcpkg
```

重新打开终端后检查：

```powershell
echo $env:VCPKG_ROOT
```

### 8.3 构建项目

在项目根目录执行：

```powershell
cmake --preset windows-debug
cmake --build --preset windows-debug
ctest --preset windows-debug
```

Release 构建：

```powershell
cmake --preset windows-release
cmake --build --preset windows-release
```

## 10. 如何保证 Windows / Linux 都能正常运行

### 10.1 统一使用 CMake target

不要写这种：

```cmake
include_directories(...)
link_directories(...)
```

优先写：

```cmake
target_link_libraries(ai_sdk
    PUBLIC
        cpr::cpr
        nlohmann_json::nlohmann_json
        spdlog::spdlog
)
```

这样 CMake 会自动处理 include 路径、lib 路径和平台差异。

### 10.2 不写平台相关网络代码

不要直接使用：

* WinHTTP
* Windows Socket API
* Linux epoll
* Linux curl 命令行

第一版统一走：

```text
AIClient
  ↓
HttpClient
  ↓
cpr
  ↓
libcurl
```

这样 Windows 和 Linux 的网络差异被 cpr/libcurl 屏蔽。

### 10.3 文件路径使用 std::filesystem

不要手写 `/` 和 `\\`。

推荐：

```cpp
#include <filesystem>

namespace fs = std::filesystem;

fs::path config_path = fs::current_path() / "config.json";
```

### 10.4 环境变量读取封装一层

API Key 不要写死在代码里。

建议封装：

```cpp
std::optional<std::string> getEnv(const std::string& name);
```

Windows 和 Linux 都可以通过环境变量传：

Windows PowerShell：

```powershell
$env:DEEPSEEK_API_KEY="your_api_key"
$env:MINIMAX_API_KEY="your_api_key"
```

Linux：

```bash
export DEEPSEEK_API_KEY="your_api_key"
export MINIMAX_API_KEY="your_api_key"
```

### 10.5 编码统一用 UTF-8

Windows 下特别注意中文输出和路径问题。

建议：

* 源码文件统一 UTF-8。
* MSVC 编译加 `/utf-8`。
* 日志和 JSON 内部统一使用 UTF-8。
* 文件路径尽量用 `std::filesystem::path`。

CMake 中：

```cmake
if (MSVC)
    target_compile_options(ai_sdk PRIVATE /utf-8)
endif()
```

### 10.6 避免平台相关系统调用

第一版不要直接写：

```cpp
system("ls");
system("dir");
```

如果后续做 Shell Tool，要做平台判断：

```cpp
#ifdef _WIN32
    // Windows command
#else
    // Linux command
#endif
```

但 SDK 核心层尽量不要依赖这些。

### 10.7 CI 双平台构建

后续可以加 GitHub Actions 或 Gitee CI，至少跑：

```text
Windows + MSVC
Linux + GCC
Linux + Clang
```

最小矩阵：

```text
windows-latest + msvc
ubuntu-latest + gcc
```

每次提交自动执行：

```bash
cmake --preset xxx
cmake --build --preset xxx
ctest --preset xxx
```

## 11. 第一版依赖结论

第一版最终依赖就定成：

```text
CMake
vcpkg
cpr
nlohmann/json
spdlog
GoogleTest
```

暂时不引入：

```text
Boost
Qt
SQLite
yaml-cpp
protobuf
stduuid
Python
```

这样项目会保持干净，跨平台难度也最低。
